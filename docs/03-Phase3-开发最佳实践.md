# Phase 3: 开发最佳实践

## 概念 1: 验证优先原则

**这是 Claude Code 官方认定的最高杠杆实践。**

### 核心思想
不要只说"帮我写个功能"，要说"帮我写个功能，验证方法是："
- 测试用例
- 截图/预期输出
- 验收标准

### 为什么验证优先？
| 做法 | 效果 |
|------|------|
| 不给验证方法 | Claude 写完就算，可能有隐藏 bug |
| 给测试/验收标准 | Claude 会自检，错了会主动改 |

### 验证的三个层次
| 层次 | 方式 | 适用场景 |
|------|------|----------|
| **自检** | Claude 自己读代码检查 | 小改动 |
| **测试** | 运行单元测试/集成测试 | 需要编译/运行的项目 |
| **人工** | 让你看截图/效果 | UI 类 |

### 实践技巧
```
给测试用例：
[1,3,2] → [1,2,3]
[5,4,3,2,1] → [1,2,3,4,5]
[] → []
```

## 概念 2: 代码质量保障

### 四个工具

| 工具 | 作用 |
|------|------|
| **Hooks** | 自动触发（保存后 lint、提交前测试）|
| **权限模式** | 限制危险操作 |
| **代码审查** | /review |
| **安全审查** | security-review |

### 权限模式

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| `default` | 每次操作都确认 | 学习阶段 |
| `plan` | 只规划不执行 | 复杂任务规划 |
| `minimal` | 禁用危险工具 | 生产环境 |
| `auto` | 自动允许所有操作 | 信任环境 |

### 常用命令
- `/review` — 代码质量审查
- `Skill: security-review` — 安全审查

## 概念 3: Bug 修复工作流

### 五步走
```
1. 粘贴错误信息（不要总结）
2. 让 Claude 追踪根因
3. 审查 diff
4. 运行测试验证
5. 提交修复
```

### 关键要点
- 粘贴完整错误信息，不要总结
- 先分析再动手，避免头痛医头
- 用 `/diff` 查看变更
- 测试通过后再提交

## 概念 4: 重构策略

### 核心原则
| 原则 | 说明 |
|------|------|
| **Plan Mode 规划** | 重构前先规划步骤 |
| **小步迭代** | 每次只改一个维度 |
| **增量式重构** | 每步都要能运行 |
| **行为不变** | 改完功能完全一致 |

### 常见代码坏味道
- 过长函数（>50 行）
- 重复代码（>3 处）
- 过深嵌套（>3 层）
- 参数过多（>5 个）
- 命名不清

### 重构流程
1. 进入 Plan Mode
2. 分析代码，识别坏味道
3. 制定重构步骤
4. 每步执行后运行测试
5. 确认测试通过后再进下一步

## 概念 5: 测试驱动开发

### 核心思路
```
1. 先写测试（描述期望行为）
2. 测试失败
3. 写代码让测试通过
4. 重构优化
```

### Claude 写测试的关键

**1. 匹配项目现有模式**
让 Claude 读现有测试文件，了解：
- 测试框架（Jest/Vitest/Mocha）
- 测试文件组织
- 断言风格
- Mock 用法

**2. 明确边界条件**
```
"测试这些边界：
- 空数组 → 返回空数组
- 单元素数组 → 返回原数组
- 包含重复元素 → 保留重复"
```

**3. 验证覆盖度**
生成测试后，人工检查核心路径、边界情况、错误处理。

## 概念 6: Git 工作流集成

### 提交信息规范（Conventional Commits）
```
<type>(<scope>): <description>
```

| type | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构 |
| `docs` | 文档 |
| `test` | 测试 |
| `chore` | 杂项 |

### 分支策略
```
main (保护)
  ↑
feature/xxx (功能分支)
```

### Worktree 并行开发
```python
# 创建新分支的 worktree
EnterWorktree({
  name: "feature-new-login"
})

# 退出
ExitWorktree({
  action: "keep"  # 保留目录
  # 或 "remove" 删除
})
```

使用场景：同时开发多个功能、实验性改动不污染主分支

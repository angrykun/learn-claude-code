# Phase 4: 高级特性实战 - 记忆系统进阶

## 概述

Claude Code 的记忆系统是其最具价值的长期功能之一。通过 Auto Memory，Claude 可以在不同会话之间记住用户的偏好、项目约束和重要信息。本章节深入讲解记忆系统的四种类型、存储结构和使用策略。

---

## 1. 记忆系统架构

### 1.1 记忆存储位置

Claude Code 的记忆存储在以下位置：

```
~/.claude/projects/<项目路径>/memory/
├── MEMORY.md          # 索引文件（前200行自动加载）
├── user/              # 用户信息记忆
├── feedback/         # 反馈规则记忆
├── project/          # 项目信息记忆
└── reference/        # 外部资源引用
```

### 1.2 记忆加载机制

- **自动加载**: MEMORY.md 的前 200 行在每个会话开始时自动加载
- **按需加载**: 子目录中的记忆文件在访问该目录时懒加载
- **手动加载**: 使用 `/memory` 命令查看和编辑所有记忆

## 2. 四种记忆类型

### 2.1 User 记忆 - 用户信息

**用途**: 存储关于用户身份、角色、偏好和背景的信息

**特征**:
- 用户明确告知的个人信息
- 长期有效的身份特征
- 协作风格的偏好

**示例**:
```markdown
---
name: user_senior_engineer
description: 用户是资深软件工程师，10年后端开发经验
type: user
---

用户角色：后端架构师
技术栈：Go、Python、Kubernetes
协作风格：喜欢直接指出问题，不喜欢绕弯子
偏好：详细的技术解释，代码示例
```

**保存时机**:
- 用户首次介绍自己的角色和经验
- 用户明确说明技术偏好（如 "我喜欢用 Go"）
- 用户说明协作风格偏好

### 2.2 Feedback 记忆 - 反馈规则

**用途**: 记录用户给出的修正和确认，用于指导未来行为

**特征**:
- 来自用户的纠正（"不要这样做"）
- 确认有效的方法（"是的，这样很好"）
- 包含原因说明（为什么这样/不这样）

**结构要求**:
```
规则本身
**Why:** (用户给出的原因)
**How to apply:** (何时应用这个规则)
```

**示例**:
```markdown
---
name: feedback_testing_strategy
description: 集成测试必须使用真实数据库，不能用 mocks
type: feedback
---

集成测试必须使用真实数据库，禁止使用 mocks。

**Why:** 上季度一次 mock 测试通过但生产迁移失败的教训

**How to apply:** 任何涉及数据库操作的测试都使用测试数据库，不使用内存 mock
```

**保存时机**:
- 用户明确纠正你 ("不要用 XXX，要用 YYY")
- 用户确认一个方法有效 ("是的，这样是对的")
- 用户给出非显而易见的偏好

### 2.3 Project 记忆 - 项目信息

**用途**: 存储项目的当前状态、目标和上下文

**特征**:
- 项目特定的约束和目标
- 进行中的工作状态
- 团队和利益相关者信息

**示例**:
```markdown
---
name: project_current_initiative
description: 当前项目是重构认证系统，计划6月完成
type: project
---

当前 initiative：重构认证系统
目标：替换旧的 session 机制为 JWT
截止日期：2026年6月
关键利益相关者：安全团队、CTO
已完成：方案设计、API 定义
进行中：用户模块重构
```

**保存时机**:
- 了解项目的当前目标和优先级
- 知道谁在做什么项目
- 项目的截止日期和约束

### 2.4 Reference 记忆 - 外部资源

**用途**: 记住外部系统的位置和访问方式

**特征**:
- 外部工具和系统的位置
- API 端点和访问凭证位置
- 文档和资源的链接

**示例**:
```markdown
---
name: reference_tracking_systems
description: 项目使用的跟踪系统和文档位置
type: reference
---

缺陷跟踪：Linear 项目 "INGEST"
监控面板：grafana.internal/d/api-latency
设计文档：Notion 数据库 engineering-docs
代码审查：GitHub PR 需经过 2 人批准
```

**保存时机**:
- 用户告诉你某个资源的位置
- 发现一个有用的外部系统
- 需要记住某个工具的访问方式

## 3. 记忆操作命令

### 3.1 /memory 命令

```bash
/memory          # 打开记忆编辑器
/memory show     # 显示当前加载的记忆
/memory add      # 添加新记忆
/memory search   # 搜索记忆内容
```

### 3.2 自然语言保存

直接在对话中声明：
- "记住：总是使用 pnpm"
- "记住我的偏好：严格 TypeScript 模式"
- "记住这个项目的 API 规范在 docs/api-spec.md"

### 3.3 显式保存格式

```markdown
---
name: <记忆名称>
description: <一行描述>
type: <user|feedback|project|reference>
---

<记忆内容>
```

## 4. 记忆最佳实践

### 4.1 何时保存

| 场景 | 记忆类型 | 示例 |
|------|---------|------|
| 用户介绍自己 | user | "我是数据科学家" |
| 用户纠正你 | feedback | "不要用 mock 测试" |
| 项目目标 | project | "Q3 完成微服务迁移" |
| 外部资源位置 | reference | "Bugs 在 Linear" |

### 4.2 何时访问

- 开始新任务时检查 project 记忆
- 收到反馈时检查 feedback 记忆
- 用户提问时检查 user 记忆
- 需要外部资源时检查 reference 记忆

### 4.3 何时忽略记忆

- 用户明确说 "忽略之前的记忆"
- 任务与记忆无关
- 记忆已过时

### 4.4 记忆质量原则

1. **原子性**: 一个记忆只包含一件事
2. **可操作性**: 包含如何应用的指导
3. **包含原因**: feedback 类型必须包含 Why
4. **可验证**: 能通过实际行为验证是否遵守

## 5. 记忆系统进阶技巧

### 5.1 记忆组织

MEMORY.md 索引文件结构：
```markdown
- [用户角色](user/user_role.md) — 10年经验的Go工程师
- [测试策略](feedback/feedback_testing.md) — 必须用真实数据库
- [当前项目](project/project_initiative.md) — 认证系统重构
- [跟踪系统](reference/reference_systems.md) — Linear 和 Grafana
```

每行不超过 150 字符，保持简洁。

### 5.2 记忆更新

- 定期检查记忆是否仍然有效
- 发现错误时更新而非删除
- 过时记忆标记为 deprecated 或删除

### 5.3 记忆与 CLAUDE.md 的区别

| 特征 | CLAUDE.md | Memory |
|------|-----------|--------|
| 加载时机 | 每个会话开始 | 记忆文件被访问时 |
| 持久性 | 静态规则 | 动态积累 |
| 更新方式 | 手动编辑 | 自然语言或手动 |
| 用途 | 项目规范 | 会话间知识 |

## 6. 实战练习

### 练习 1: 保存用户记忆（5分钟）

1. 告诉 Claude 你的角色和技术背景
2. 观察 Claude 是否正确保存为 user 类型记忆
3. 使用 `/memory` 验证保存结果

### 练习 2: 保存反馈记忆（10分钟）

1. 给出一个非显而易见的偏好（如 "不要在代码中留 TODO 注释"）
2. 给出原因（Why）
3. 验证保存结果

### 练习 3: 创建项目记忆（10分钟）

1. 告诉 Claude 一个当前项目目标
2. 包含截止日期和关键约束
3. 验证保存结果

### 练习 4: 记忆检索（5分钟）

1. 使用 `/memory search <关键词>` 搜索记忆
2. 验证检索结果的准确性

### 练习 5: 记忆清理（5分钟）

1. 检查现有记忆
2. 删除过时或错误的记忆
3. 更新需要修正的记忆

## 7. 常见问题

### Q1: 记忆会永久保存吗？

是的，记忆保存在 `~/.claude/projects/` 目录下，除非手动删除或清除项目数据。

### Q2: 记忆会影响上下文大小吗？

会，但记忆是按需加载的。MEMORY.md 前 200 行自动加载，其他记忆只在相关时才加载。

### Q3: 可以跨项目共享记忆吗？

不能直接在项目间共享，但可以通过复制记忆文件到其他项目的 memory 目录。

### Q4: 记忆安全吗？

记忆存储在本地文件系统，与项目代码分开。除非有人物理访问你的电脑，否则是安全的。

## 8. 本章小结

- **四种记忆类型**: user（用户）、feedback（反馈）、project（项目）、reference（引用）
- **保存时机**: 用户纠正、确认有效方法、项目决策、外部资源位置
- **访问时机**: 相关任务、用户明确要求
- **质量原则**: 原子性、可操作性、包含原因、可验证
- **命令**: `/memory` 查看/编辑，`记住:` 声明式保存
